# Введение {#введение .unnumbered}

Телехирургия --- это современная медицинская практика, которая
объединяет хирургические процедуры с использованием телероботов и
дистанционного управления.

Во время телехирургии хирург использует специальный роботический
систему, состоящую из консоли управления и манипуляторов, чтобы
выполнять операции на пациенте. Хирург сидит за консолью и управляет
манипуляторами, которые передают движения на инструменты, введенные в
тело пациента через небольшие разрезы. Камера передает изображение
высокой четкости на консоль, позволяя хирургу видеть операционное поле в
реальном времени.

Телехирургия предоставляет несколько преимуществ. Она позволяет хирургам
осуществлять сложные и точные движения с высокой маневренностью робота.
Кроме того, она может предложить более точные результаты, менее
инвазивные процедуры и более быстрое восстановление для пациентов.

Одним из самых известных применений телехирургии является роботическая
ассистированная хирургия (РАХ), которая используется в различных
областях, таких как кардиохирургия, урология, гинекология и
гастроэнтерология. РАХ позволяет хирургам делать точные и сложные
операции с помощью роботических манипуляторов. Однако, важно отметить,
что телехирургия имеет свои ограничения. В некоторых случаях, например,
при реализации операций с открытым доступом или в случае экстренных
операций, телехирургия может быть менее эффективной. В таких ситуациях
традиционные методы хирургии могут предпочтительнее.

В целом, телехирургия представляет собой инновационный подход к
хирургическому лечению, который продолжает развиваться и улучшаться. Она
обещает предоставить хирургам больше возможностей для точных и
эффективных процедур, а пациентам - более безопасное и быстрое
восстановление.

Данное руководство предназначено для ознакомления с роботизированной
системой для проведения эндоваскулярной хирургии в целях выполнения
лабораторных работ. В нём описаны основные элементы роботизированной
системы, а также принцип управления роботами-манипуляторами и их режимы
работы. Также приведён исходный код программ управления.

В текущий момент алгоритм для управления роботами-манипуляторами
постоянно обновляется. Руководство основано на актуальной версии
алгоритма, при этом перечислены недостатки текущей версии и планируемые
изменения в структуре алгоритма.

# 3 Контроллер в программном обеспечении {#контроллер-в-программном-обеспечении .unnumbered}

Связь робота с контроллером происходит при помощи программы
Joystick_hirurg.py (Код программы приведён в П.A1).

При старте программы создаётся объект класса Robot, конструктор которого
принимает в качестве аргумента строку, в которой содержится ip-адрес для
установления соединения с роботом по ethernet кабелю. Далее создаётся
объект класса Service, конструктор которого принимает в качестве
аргумента 3 вещественных числа: линейную скорость, скорость вращения и
ускорение, после чего вызывается метод init_joystick(). Затем в
конструкции try-except вызывается метод loop(), а в случае возникновения
ошибки -- метод close()

Основной код программы реализован в классе Service. Данный класс
содержит в себе следующие переменные:

-   Объект класса Joystick, который инициализирует контроллер. В
    конструктор передаётся целочисленное число -- номер контроллера.
    Контроллеру хирурга соответствует число 0, контроллеру
    робота-диагноста -- 1;

-   Объект класса Robot, который был создан при старте программы;

-   В случае с программой для хирурга также создаётся объект класса
    Robot, в конструктор которого передаётся ip-адрес робота-диагноста;

-   Линейная скорость движения робота;

-   Скорость вращения робота;

-   Ускорение робота;

-   Объект класса Cmd, который хранит в себе состояния кнопок и
    отклонения по осям джойстика.

-   В случае с программой для хирурга, логическая переменная отвечающая
    за режим управления.

Состояния кнопок 0-11, а также отклонения джойстика 12 и миниджойстика
13 по осям хранятся в объекте класса Cmd (рис. 5). Где axis0 -- вращение
джойстика 12 вокруг своей оси, axis1 соответствует отклонению джойстика
12 по оси X, axis1 соответствует отклонению джойстика 12 по оси Y,
данные переменные принимают значения в диапазоне \[-1,1\], при этом
значение 0 соответствует центральному положению. Переменные btn0 -- btn9
хранят состояния кнопок 0-9 соответственно, принимают значения 0 --
кнопка не нажата или 1 -- кнопка нажата. Список hat0 хранит отклонение
миниджойстика по его двум осям \[X, Y\] данные в списке могут принимать
значения -1, 0, 1, при этом 0 -- центральное положение на оси.

Класс Service содержит в себе следующие методы:

-   Метод init_joystick()

Данный метод предназначен для инициализации контроллера. В ходе его
выполнения происходит старт модуля pygame для работы с джойстиками и
создание объекта класса Joystick.

-   Метод loop()

Данный метод представляет из себя бесконечный цикл, в ходе которого
происходит обнуление состояния кнопок и положения контроллера с помощью
метода reset() класса Cmd, обработка событий pygame, получение текущего
положения робота, получение информации об отклонениях по осям джойстика
и состояния кнопок, и их запись в соответствующую переменную объекта
класса Cmd, создание массива скоростей робота, запись в массив значения
скоростей исходя из заданной линейной скорости и скорости вращения и
отклонениях по осям джойстика и состояний кнопок 2 и 3, затем полученный
массив скоростей передаётся роботу, после чего он начинает движение. В
программе для хирурга происходит получение текущей матрицы перехода от
системы координат, связанной с основанием робота, к системе координат,
связанной с рабочим окончанием робота. С помощью данной матрицы задаются
проекции скоростей на плоскость стола в случае, если активен режим
синхронного движения. При активации режима асинхронного режима движения
программа для управления роботом-диагностом запускается в отдельном
потоке, включение режима синхронного движения приводит к завершению
программы контроллера робота-диагноста посредством прерывания
бесконечного цикла, что приводит к вызову метода close() класса Service.

-   Метод close() -- завершает работу с контроллером.

Для упрощения кода был создан более абстрактный алгоритм управления с
помощью контроллеров, основной особенностью которого является
определения роли робота-манипулятора- master или slave, то есть хирург
или диагност, при этом если робот-манипулятор является мастером, то он
также ожидает получить IP слейва при инициализации. При этом в данном
алгоритме отсутствует класс cmd, вместо этого для увеличения
быстродействия в необходимый момент происходит считывание только
необходимых в данный момент кнопок контроллера через встроенные методы
модуля pygame -- joystick.get_button() и joystick.get_axis().

Таким образом обновлённый алгоритм обладает только одним классом Robot,
который обладает следующими методами:

-   \_\_init\_\_(self, IP, is_master, logger, auto, lock, shared_path,
    heartbeat, slave_IP=None) -- конструктор класса, который принимает
    следующие переменные:

    -   IP -- адресс робота-манипулятора

    -   Is_master -- флаг указывающий на роль робота манипулятора

    -   Logger -- логгер (см. раздел 6.4)

    -   Auto -- флаг синхронного режима перемещения (см. раздел 4.1)

    -   Shared_path -- массив для записи траектории робота (см. раздел
        4.2)

    -   Heartbeat -- очередь для записи состояния процесса (см. раздел
        6.5)

    -   Slave_IP -- адрес слейва, нужен только для мастера

-   Init_robot(self) -- инициализирует подключение к
    роботам-манипуляторам

-   Init_joystick(self) -- инициализирует подключение к джойстикам

-   set_robot_settings(self, linear_velocity=0.3,
    rotational_velocity=0.1, acceleration=0.1) -- устанавливает
    следующие настройки робота-манипулятора:

    -   linear_velocity -- скорость линейного перемещения в м/с

    -   rotational_velocity -- скорость вращения в рад/с

    -   acceleration -- ускорение в м/с^2^

-   main_loop(self) -- аналогичен вышеописаному методу Loop()

-   start_robot(self) -- последовательно выполняет вышеперечисленные
    методы

-   get_speeds(self) -\>speeds\[6\] -- метод вычисления массива
    скоростей робота-манипулятора, вызывается в main_loop

-   get_slave_speeds(self)-\>speeds\[6\] -- аналогично для слейва

-   update_heartbeat(self) -- обновляет сердцебиение (см. раздел 6.5),
    выполняется в main_loop

-   close(self) -- закрывает все соединения

# 4 Режимы управления роботами-манипуляторами {#режимы-управления-роботами-манипуляторами .unnumbered}

В программе реализовано два режима управления роботом: режим синхронного
и асинхронного движения манипуляторов. При синхронном режиме управление
посредством контроллера осуществляется только над хирургом, при этом
робот-диагност движется в одном направлении с хирургом автоматически, а
его контроллер отключён. При данном режиме робот-диагност движется в
фиксированной плоскости, параллельной плоскости стола, при этом проекции
скоростей двух роботов на плоскость стола совпадают. При асинхронном
режиме управление каждым роботом производится по отдельности
соответствующим контроллером.

## 4.1 Управление роботами с помощью контроллеров (РПК) {#управление-роботами-с-помощью-контроллеров-рпк .unnumbered}

Кнопкам 6-8 контроллера хирурга приписаны следующие функции:

-   Кнопка 6 переключает режим управления в синхронный режим;

-   Кнопка 7 переключает режим управления в асинхронный режим;

-   Кнопка 8 переводит робота-диагноста в плоскость параллельную
    плоскости стола.

Управление с помощью джойстика 12 и миниджойстика 13 идентично для обоих
роботов. При отклонении джойстика 12 от себя (направление Y (рис.2))
происходит поворот конечного звена робота против часовой стрелки вокруг
оси Y стола (рис. 4б), при отклонении на себя (направление -Y (рис.2))
происходит поворот конечного звена робота по часовой стрелки вокруг оси
Y стола (рис. 4в).

V'_{y} = x_{y}V_{x} + y_{y}V_{y} + z_{y}V_{z} \\
V'_{z} = 0
\end{array} \right.\ $$

Где V'~n~ -- скорость робота-диагноста по оси n, V~k~ -- скорость
робота-хирурга по оси k

## 6.2 Алгоритм контроля постоянного силового давления

За основу алгоритма КПСД была выбрана следующая модель взаимодействия
робота-манипулятора с биообъектом.

$$\begin{array}{r}
{\dot{x}}_{d} = - k_{f}\left( f_{c} - f_{d} \right)
\end{array}$$

Где k~f~ -- коэффициент податливости, параметр скорости корректировки
силового давления

f~c~ -- текущее значение силового давления

f~d~ -- целевое значение силового давления

ẋ~d~ -- скорость робота-манипулятора

Поскольку воздействие происходит по оси z:

$$\begin{array}{r}
v_{z} = - k_{f}\left( f_{z} - f_{d} \right),
\end{array}$$

где v~z~ -- скорость по оси z, которую необходимо передать роботу для
корректировки силового давления;

f~z~ -- регистрируемое силовое давление по оси z.

Для компенсации воздействия веса инструмента и движения
робота-манипулятора введём f~comp~, определяемое по усреднённому
показанию датчика силового воздействия по оси z при движении конечного
звена, не касаясь объектов.

Итоговое уравнение движения:

$$\begin{array}{r}
v_{z} = - k_{f}\left( \left( f_{z} - f_{comp} \right) - f_{d} \right)
\end{array}$$

В случае если учитывается силовое воздействие по всем осям, то вместо fz
используется суммарный вектор f = (fx,fy,fz)^T^, при этом корректировка
скорости остаётся всё также по оси z.

6.2.1 Описание алгоритма и его реализация

На основе (21) был составлен алгоритм КПСД:

1.  Установка параметров:

    1.  v~zmax~ -- максимальная скорость корректировки в м/с;

    2.  k~f~ -- коэффициент податливости, параметр, отвечающий за то,
        насколько быстро робот-манипулятор будет адаптироваться;

    3.  f~d~ -- силовое давление, которое должно сохраняться постоянным
        в Н.

2.  Калибровка датчика силового воздействия «на весу»:

    1.  Конечное звено устанавливается непосредственно над стартовой
        позицией, но не касаясь никаких поверхностей;

    2.  Конечное звено перемещается в течении 5с в сторону, при этом всё
        время происходит считывание показаний датчика силового давления;

    3.  Возврат конечного звена в исходную точку;

    4.  Усреднение показаний силового давления -- расчёт f~comp~;

3.  Конечное звено начинает движение вниз, до достижения f~d~;

4.  Перемещение конечного звена в заданном направлении, используя
    управление скоростью:

    1.  Считывание f~z~;

    2.  Расчёт v~z~ по формуле (21);

    3.  Если \|v~z~\| \> \|v~zmax~\|, то v~z~ = v~zmax~ по модулю,
        сохраняя при этом исходный знак;

    4.  Составление нового вектора скорости V = (v~x~, v~y~, v~z~)^T^,
        при этом v~x~, v~y~ данным алгоритмом не изменяются;

    5.  Передача вектора скорости на блок управления
        робота-манипулятора;

5.  Пункт 4 повторяется до остановки алгоритма извне.

Разработанный алгоритм был реализован на языке программирования Python
3.8 с использованием модуля для управления роботом-манипулятором модели
URX. Для осуществления управления используется объект класса robot,
который обладает следующими методами:

1.  robot.getl() -- возвращает текущее положение конечного звена в
    пространстве;

2.  robot.movel(pose,acc, vel) -- метод перемещения в положение в
    пространстве, описываемое матрицей pose с ускорением acc м/с^2^ и
    скоростью vel м/с;

3.  robot.speed_tool(V, acc, min_time) -- метод перемещения конечного
    звена с заданным вектором скорости V = \[vx, vy, vz, wx, wy, wz\],
    где vx, vy, vz -- скорости по координатным осям x y z соответственно
    в м/с, wx, wy, wz -- скорость поворота в рад/с вокруг
    соответствующих осей;

4.  robot.get_tcp_force() -- возвращает показания датчика силового
    давления в виде списка F = \[Fx, Fy, Fz, Mx, My, Mz\], где Fi --
    усилие по оси i в Н, Mi -- момент на оси i в Н/м;

5.  Остальные методы класса не применяются в данной реализации алгоритма
    КПСД, поэтому не рассматриваются.

Пункты 1-2 алгоритма были реализованы в функции:

def force_calibration(rob, vel, acc, cal_time):

Fcomp = \[\]

start_pose = rob.getl()

rob.speedl_tool(\[vel,0,0,0,0,0\], acc, cal_time)

start_time = time.time()

while (time.time() - start_time) \< cal_time:

Fs.append(rob.get_tcp_force()\[2\])

rob.speedl_tool(\[0\] \* 6, 0.5, 2)

rob.stopl()

Fs = sum(Fs)/len(Fs)

rob.movel(start_pose, acc = 0.05, vel = 0.05)

return Fcomp

Функция принимает на вход объект робота скорость, при которой будет
происходить калибровка в м/с, ускорение робота в м/с^2^ и время, в
течении которого будет происходить усреднение показаний датчика силового
давления. Возвращает значение компенсации усилия.

Пункт 3 реализован в функции:

## 6.5 Алгоритм отслеживания статусов процессов

Поскольку процессы исполняются в отдельных областях программы, для
отслеживания ошибок, возникающих в них, был разработан watchdog, для их
контроля и перезапуска случае необходимости.

def watchdog(self):\
logger.debug(\"Запущен цикл мониторинга сердцебиения\")\
while True:\
current_time = time.time()\
while not self.heartbeat.empty():\
msg = self.heartbeat.get()\
name = msg\[0\]\
if name not in self.processes:\
logger.debug(f\"Получено сообщение от неизвестного процесса {name}:
{msg}\")\
continue\
\
proc = self.processes\[name\]\
msg_type = msg\[1\]\
\
if msg_type == \"READY\":\
proc\[\'state\'\] = \"READY\"\
proc\[\'last_heartbeat\'\] = msg\[2\]\
logger.info(f\"Процесс {name} (PID: {proc\[\'pid\'\]}) готов к
работе\")\
self.\_update_status(name, \"READY\", \"lightgreen\")\
\
elif msg_type == \"ALIVE\":\
prev_state = proc\[\"state\"\]\
proc\[\"state\"\] = \"RUNNING\"\
proc\[\"last_heartbeat\"\] = msg\[2\]\
\
if prev_state != \"RUNNING\" or (current_time - getattr(proc,
\"last_log_time\", 0)) \> 30:\
uptime = current_time - proc\[\"start_time\"\]\
logger.debug(\
f\"Процесс {name} активен \| PID: {proc\[\'pid\'\]} \| Аптайм:
{uptime:.1f}s \| Последнее сердцебиение: {current_time -
proc\[\'last_heartbeat\'\]:.2f}s назад\")\
proc\[\"last_log_time\"\] = current_time\
\
self.\_update_status(name, \"RUNNING\", \"green\")\
\
elif msg_type == \"ERROR\":\
error_msg = msg\[2\]\
logger.error(f\"Процесс {name} (PID: {proc\[\'pid\'\]}) ошибка:
{error_msg}\")\
self.\_force_restart(name, reason=f\"Ошибка: {error_msg}\")\
\
elif msg_type == \"CRASH\":\
error_msg = msg\[2\]\
logger.critical(f\"Процесс {name} (PID: {proc\[\'pid\'\]}) аварийно
завершился: {error_msg}\")\
self.\_force_restart(name, reason=f\"Аварийное завершение:
{error_msg}\")\
\
for name in list(self.processes.keys()):\
proc = self.processes\[name\]\
time_since_hb = current_time - proc\[\"last_heartbeat\"\]\
\
if time_since_hb \> self.heartbeat_timeout:\
if not proc\[\'process\'\].is_alive():\
logger.warning(\
f\"Процесс {name} (PID: {proc\[\'pid\'\]}) завершился без уведомления.
Последнее сердцебиение: {time_since_hb:.1f}s назад\")\
self.\_force_restart(name, reason=\"Неожиданное завершение процесса\")\
else:\
logger.critical(\
f\"ОБНАРУЖЕНО ЗАВИСАНИЕ: процесс {name} (PID: {proc.get(\'pid\',
\'N/A\')}) не отвечает \"\
f\"в течение {time_since_hb:.1f}s (тайм: {self.heartbeat_timeout}s) \"\
f\"Состояние: {proc\[\'state\'\]}\"\
)\
self.\_force_restart(name, reason=\"Зависание процесса (таймаут
сердцебиения)\")\
\
time.sleep(1.0)

Данный метод запускается отдельным потоком при старте ЕППУИХ. Каждый
процесс периодически кладёт в общую очередь heartbeat своё состояние с
отметкой по времени, состояния могут быть следующими:

-   STARTING -- процесс находится на этапе инициализации

-   ALIVE -- процесс работает исправно

-   ERROR -- процесс завершился с ошибкой

-   CRASH -- в процессе произошёл критический сбой

-   AWAITING -- процесс ожидает запуска

-   FINISHED -- процесс завершён или ожидает завершения извне

Так же watchdog мониторит сердцебиение процессов, то есть количество
времени, прошедшее с момента последнего обновления статуса.

Если watchdog получает сообщение от процесса со статусом ERROR или CRASH
или процесс не обновлял сердцебиение слишком долго, то он вызывает метод
планировки перезапуска проблемного процесса:

def \_force_restart(self, name, reason=\"Неизвестная причина\"):\
proc = self.processes\[name\]\
if not proc:\
logger.warning(f\"Попытка перезапуска несуществующего процесса
{name}\")\
return\
\
p = proc\[\"process\"\]\
pid = proc.get(\'pid\', \'N/A\')\
\
logger.warning(f\"Начало принудительного перезапуска {name} (PID: {pid})
\| Причина: {reason}\")\
\
if p.is_alive():\
logger.debug(f\"Отправка SIGTERM процессу {name} (PID: {pid})\")\
p.terminate()\
p.join(timeout=3.0)\
\
if p.is_alive():\
logger.error(f\"Процесс {name} (PID: {pid}) не отвечает на SIGTERM,
отправка SIGKILL\")\
p.kill()\
p.join(timeout=2.0)\
\
if p.is_alive():\
logger.critical(f\"НЕВОЗМОЖНО ЗАВЕРШИТЬ процесс {name} даже после
SIGKILL!\")\
else:\
logger.info(f\"Процесс {name} (PID: {pid}) успешно завершён\")\
\

del self.processes\[name\]\
\
delay = self.restart_delays.get(name, 1.0)\
self.restart_delays\[name\] = min(delay \* 2, 30.0)\
\
logger.info(f\"Планирование перезапуска {name} через {delay:.1f}с
(экспоненциальная задержка)\")\
\
threading.Timer(delay, self.\_actual_restart, args=(name,
proc\[\'target\'\], proc\[\'params\'\], reason)).start()

Данный метод пытается принудительно завершить процесс при его зависании
(если это необходимо), после чего планирует перезапуск с
экспоненциальной задержкой вызывая метод перезапуска:

def \_actual_restart(self, name, target_func, params, reason=\"\"):\
self.restart_delays\[name\] = 1.0\
\

\
p = mp.Process(target=target_func, args=params, daemon=True, name=name)\

\

\'process\': p,\
\'last_heartbeat\': time.time(),\
\'start_time\': time.time(),\
\'params\': params,\
\'state\': \'STARTING\',\
\'pid\': p.pid,\
\'target\': target_func\

Данный метод пересоздаёт объект процесса, после чего перезапускает его,
в случае неудачи, watchdog поймает снова незапустившийся процесс, после
чего данный метод будет вызван снова, но с увеличенной задержкой

Для запуска процессов основная программа добавляет в очередь имя
процесса со статусом AWAITING, после чего алгоритм watchdog

## 6.6 Алгоритм калибровки роботов манипуляторов

Для грамотной координации роботов-манипуляторов необходим алгоритм
калибровки, который позволит одному роботу-манипулятору определять
положение второго относительно себя.

Имеется две системы координат:

-   Система 1 (Base1) -- система координат основания первого
    робота-манипулятора

-   Система 2 (Base2) -- система координат основания второго
    робота-манипулятора

Необходимо найти однородную матрицу преобразования T, которая переводит
координаты из base1 в base2:

$$\begin{array}{r}
P_{2} = T\left( P_{1} \right) = R \times P_{1} + t\ \#(6)
\end{array}$$

Где

P~1~ = \[x1,y1,z1\]^T^ -- координаты точки в base1

P~2~ = \[x2,y2,z2\]^T^ -- координаты точки в base2

R∈ℝ^3x3^ -- матрица поворота (ортогональная, det(R) = 1)

t∈ ℝ^3^ -- вектор переноса

### **6.6.1 Калибровка по общим точкам** {#калибровка-по-общим-точкам .unnumbered}

Наиболее простым методом калибровки будет использование общих точек. На
каждом роботе-манипуляторе закрепляется указка известной длины 125мм
(рисунок 25), а на столе размещается n\>3 реперных точек.

![](media/media/image27.png){width="3.2204461942257216in"
height="5.212964785651794in"}

Рисунок 25 -- Калибровочная указка

Округлая форма обусловлена невозможностью идеально точно свести указки в
одну и ту же точку.

После чего к каждой точке подводятся оба робота-манипулятора в режиме
свободного перемещения (рукой). При этом для лучшей точности калибровки
лучше всего изменять ориентацию роботов-манипуляторов. В итоге
собираются {P~1~^(i)^}^n^~i=1~ и {P~2~^(i)^}^n^~i=1~, где P~1~^(i)^ и
P~2~^(i)^ -- координаты одной и той же реперной точки в системе base1 и
base2 соответственно с помощью выполнения метода urx.Robot.getl() на
каждом из роботов-манипуляторов.

Далее оптимальный переход вычисляется методом Кабша:

1.  Вычисление центроид

$$\begin{array}{r}
c_{1} = \frac{1}{n}\sum_{i = 1}^{n}P_{1}^{(i)},\ c_{2} = \frac{1}{n}\sum_{i = 1}^{n}P_{2}^{(i)}\ \#(7)
\end{array}\ $$

2.  Центрирование точек

$$\begin{array}{r}
Q_{1}^{(i)} = P_{1}^{(i)} - c_{1},\ Q_{2}^{(i)} = P_{2}^{(i)} - c_{2}\ \#(8)
\end{array}$$

3.  Вычисление матрицы ковариации H∈ℝ^3x3^

$$\begin{array}{r}
H = \ \sum_{i = 1}^{n}{Q_{1}^{(i)} \times \left( Q_{2}^{(i)} \right)^{T}} = \ Q_{1}^{T} \times Q_{2}\ \#(9)
\end{array}$$

4.  Сингулярное разложение

$$\begin{array}{r}
H = U \times \Sigma \times V^{T}\ \#(10)
\end{array}$$

Где U, V ∈ ℝ^3x3^ -- ортогональные матрицы

Σ∈ ℝ^3x3^ -- диагональная матрица сингулярных чисел

Оптимальная матрица поворота R вычисляется как:

$$\begin{array}{r}
R = \ V \times U^{T}\ \#(11)
\end{array}$$

При этом производиться проверка, если det(R) = -1, то это отражение, а
не поворот и необходимо исправление:

$$\begin{array}{r}
V^{'} = V \times diag(1,1, - 1)\#(12)
\end{array}$$

После чего (12) подставляется в (11)

Вектор переноса:

$$\begin{array}{r}
t = c_{2} - R \times c_{1}\#(13)
\end{array}$$

Обладая матрицей T можно определить расстояние между конечными звеньями
для оценки безопасности перемещения в ту или иную точку, путём
подставления в (6) координаты конечного звена робота-манипулятора,
относительно которого ведётся расчёт.

## 7.2 Организация процессов

Для задач телехирургии необходимо минимальное время отклика системы и
синхронность выполнения процессов. Поэтому в данном программном
обеспечении были использованы следующие модули:

-   Threading -- для создания отдельного потока для выполнения функции

-   Multiprocessing -- для открытия отдельного файла программы в новом
    процессе

Разница между потоком и процессом заключается в следующем:

-   Процесс -- выполняется в отдельной области памяти и имеет
    собственный интерпретатор, исполняется на отдельном ядре процессора.

-   Поток выполняется в той же области памяти, что и породивший его
    процесс, обладает тем же интерпретатором, поэтому из-за GIL (Global
    Interpreter Lock) возникает гонка за ресурсы процессора, поскольку
    только одна команда может исполняться единовременно.

Для обмена данными между процессами были созданы специализированные
разделённые переменные с помощью подмодуля manager модуля
multiprocessing, который позволяет создать объект менеджера,
управляющего серверным процессом, принцип работы которого изображён на
рисунке 25.

![](media/media/image28.png){width="6.496527777777778in"

Рисунок 25 -- Принцип обмена и синхронизации памяти между процессами

Разные процессы Python не могут напрямую записывать или читать одну и ту

переменной.

Прокси‑объекты используются в коде программы точно так же, как исходные
объекты, и логика их работы скрыта от программиста. Они сериализуют
данные, синхронизируют и координируют изменения, вносимые процессами,
с централизованной версией объекта, размещенной в менеджере.

Были созданы следующие общие переменные:

-   Program_lock -- флаг запрещающий или разрешающий перемещение
    роботов-манипуляторов, к нему обращаются процессы управления
    роботами-манипуляторами с помощью контроллеров, ППУИ и алгоритм
    контроля силового давления;

-   Auto -- флаг перехода или выхода из режима синхронного перемещения,
    используют программы алгоритма РПК, для передачи управления
    роботом-диагностом роботу-хирургу;

-   Rob_data -- массив для хранения данных телеметрии
    роботов-манипуляторов, используется в алгоритме управления
    роботом-диагностом

-   Heartbeat -- очередь используемая всеми процессами, для
    периодического отправления своего статуса
